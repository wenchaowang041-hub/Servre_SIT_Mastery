## 一、Notify 脚本 slot_of_disk 三次修复

### 1. 第一次修复：硬编码 nvme0
- **根因**：`slot_of_disk` 路径中硬编码了 `device/nvme0`，导致非 nvme0 磁盘（nvme1~11）路径不存在
- **修复**：从磁盘名动态提取控制器名（`nvme10n1` → `nvme10`）

### 2. 第二次修复：readlink -f 正则提取 BDF
- **根因**：`device/nvmeX` 路径末尾多了一层同名目录（`.../nvme/nvme0/nvme0`），`basename dirname` 取不到 BDF
- **修复**：改为 `readlink -f device` 后用正则 `([0-9a-f]{4}:[0-9a-f]{2}:[0-9a-f]{2}).X` 提取 BDF

### 3. 第三次修复：误匹配 Root Port BDF（最终方案）
- **根因**：resolved 路径包含两个 BDF（Root Port `06:04.0` 和设备本身 `09:00.0`），正则匹配的是第一个（Root Port），无法与 slot address 匹配
- **修复**：截断到 `/nvme/` 之前的目录，取其 basename 作为设备 BDF
- **验证结果**：11 块 DUT 盘全部匹配成功
  ```
  nvme0n1  -> 0-2    nvme1n1  -> 0-13   nvme2n1  -> 0-3
  nvme3n1  -> 0-12   nvme4n1  -> 0-6    nvme6n1  -> 0
  nvme7n1  -> 0-15   nvme8n1  -> 0-9    nvme9n1  -> 0-1
  nvme10n1 -> 0-14   nvme11n1 -> 0-8
  ```

## 二、编写调试脚本 debug-slot-mapping.sh

- 一键收集所有 PCI Slot（address/power）和所有 NVMe 磁盘的 device 路径
- 同时输出 readlink、readlink -f、uevent 三种方式获取的 PCI 信息
- 用于排查 slot 映射失败根因

## 三、修复 3-md5.sh 根分区爆满问题

- **现象**：dd 写到第 9 块盘（nvme9）时报 `No space left on device`
- **根因**：旧版 `dd` 先写到本地 `./md5/` 目录再 `cp` 到挂载点，11块盘 × 1GB 把根分区写满（69G 100%）
- **修复**：`dd` 直接写到 `/mnt/${name}p1/` 挂载点，不再占用本地空间

## 四、优化 Notify 脚本操作提示

## 六、光口灵活网卡无法使用问题排查

### 1. 问题现象

| 槽位 | 光口灵活网卡 | 电口灵活网卡 |
|------|-------------|-------------|
| Slot 1（上） | DOWN，不亮灯 | 正常 |
| Slot 2（中） | DOWN，不亮灯 | DOWN，不亮灯 |

### 2. 交叉验证

- 华为 SF223D-H 光口灵活网卡换到 x20 白牌机 → 网口 UP、亮灯、功能正常 → **卡本身无故障**
- 光纤、光模块经确认均正常，不存在损坏异常

### 3. 分析

- **Slot 1 电口正常但光口 DOWN** → 该槽位 PCIe 总线正常，问题出在光口适配层面（可能是 BIOS 固件对灵活网卡光口部分的 SerDes 配置不支持）
- **Slot 2 电口+光口全部 DOWN** → 该槽位本身有问题（非光口专属），可能是 PCIe 通道未启用、供电不足或 BIOS 配置错误

### 4. 后续建议

- 检查 BIOS 中 Slot 2 的 PCIe 配置是否启用
- 对比 x20 白牌机与当前机器的 BIOS 版本差异
- `lspci -vvv` 对比光口卡在两机器上的输出
- 可能需要华为原厂支持，涉及 BIOS 固件对灵活网卡光口 SerDes 的配置支持
