# Day16：系统日志与驱动核查

## 对齐计划表

- 阶段：Phase1-系统
- 周次：W3
- 模块：日志/驱动
- 主题：系统日志与驱动核查

## 今日目标

- 学会用日志和驱动信息做“证据型排障”
- 理解为什么设备异常时不能只看现象
- 建立 `日志 -> 驱动 -> 设备 -> 结论` 的排查顺序

## 先记住一句话

排障不是先猜原因，而是先找证据。

## 一、为什么 Day16 紧接 Day15

Day15 讲的是“问题分层”。

Day16 讲的是“怎么拿证据证明它属于哪一层”。

你现在要从：

- “我感觉像驱动问题”

升级成：

- “日志里能看到驱动加载、初始化、报错或重置的证据”

## 二、今天最重要的三个命令

```bash
dmesg
lsmod
journalctl -k
```

### 它们分别解决什么问题

#### `dmesg`

看内核启动和设备初始化信息。

适合看：

- 驱动加载
- 设备枚举
- 报错
- reset
- timeout

#### `lsmod`

看当前内核模块是否已经加载。

适合看：

- 某个驱动有没有加载
- 驱动依赖关系

#### `journalctl -k`

看内核日志的更完整时间线。

适合看：

- 当前启动周期内的内核消息
- 错误和时间顺序

## 三、为什么日志这么重要

因为你现场最常见的误判就是：

- 设备没出来，就说硬件坏了
- 性能差，就说链路有问题
- 工具没识别，就说卡不支持

但日志经常会直接告诉你：

- 驱动没加载
- 固件版本不匹配
- 设备训练失败
- PCIe 报错
- reset 反复发生

这时候如果你不看日志，只凭感觉，很容易走偏。

## 四、今天你该怎么用这三个命令

### 第1步：先确认模块在不在

```bash
lsmod
```

比如你关心：

- 网卡驱动
- NVMe 驱动
- NPU 驱动

那就先看对应模块有没有加载。

### 第2步：再从 `dmesg` 找关键字

你不需要一上来把整份日志全读完，可以先按设备关键字搜：

```bash
dmesg | grep -i nvme
dmesg | grep -i mlx
dmesg | grep -i hns
dmesg | grep -i pcie
dmesg | grep -i error
```

核心目的是：

- 看初始化是否成功
- 看有没有错误
- 看有没有 reset / timeout

### 第3步：必要时用 `journalctl -k` 按时间线看

```bash
journalctl -k
```

如果你发现：

- 某设备一开始正常
- 后面又报错
- 或者重置、掉线、恢复

那就不能只看单条日志了，要按时间线串起来。

## 五、今天要建立的排障顺序

以后你看到设备异常，建议固定按这个顺序：

1. 先确认设备在不在
2. 再确认驱动在不在
3. 再查日志有没有初始化和报错
4. 最后再下结论

这和“先猜一个原因再去找证据”是完全不同的。

## 六、典型例子

### 例子1：设备在，但工具不认

可能路径是：

- PCIe 已枚举
- 驱动没正常加载
- 或工具链层有问题

这时就不能直接说“卡坏了”。

### 例子2：设备一开始正常，后来掉了

这时就要重点看：

- `dmesg`
- `journalctl -k`

因为很可能存在：

- reset
- timeout
- AER 报错
- 驱动异常恢复

### 例子3：模块根本没加载

如果 `lsmod` 里没有对应模块，那排障重点就应该先回到：

- 驱动安装
- 依赖
- 内核版本兼容性

## 七、和你当前工作怎么连接

你现在已经在做：

- PCIe 设备识别
- 网卡 bring-up
- NPU 安装
- BMC 和健康状态检查

那 Day16 的作用就是：

**把“看现象”升级成“用日志和驱动做证据链”。**

以后你再说问题时，就不只是：

- “设备没识别”

而是：

- “PCIe 层可见，但驱动未加载”
- “驱动已加载，但日志中有 reset”
- “工具层未识别，但内核日志没有设备初始化成功记录”

这就是工程化表达。

## 八、今天建议输出什么

按计划表，今天建议输出：

- 日志摘录 + 解释

你可以按这个结构写：

| 日志来源 | 关键字 | 现象 | 初步解释 |
|---|---|---|---|
| dmesg | nvme | 初始化成功/失败 | NVMe 驱动和设备关系 |
| journalctl -k | pcie | reset / error | PCIe 层异常 |
| lsmod | mlx5_core | 模块存在/不存在 | 网卡驱动是否加载 |

## 九、面试式自测题

### 题1

为什么排障时不能先靠感觉判断？

标准答案方向：

- 因为现象可能来自多层，必须先用日志和驱动信息建立证据链

### 题2

`dmesg`、`lsmod`、`journalctl -k` 分别适合看什么？

标准答案方向：

- `dmesg`：初始化和报错
- `lsmod`：模块加载状态
- `journalctl -k`：内核时间线

### 题3

Day16 最核心的能力是什么？

标准答案方向：

- 从现象走到证据，而不是从现象直接跳到结论

## 十、今天的补习结论

- Day16 的重点不是背日志命令，而是建立证据型排障习惯。
- 你要从“我怀疑”升级到“日志里可以证明”。
- 这一步一旦形成，后面做整机 SIT、NPU 排障、网络和存储异常分析都会更扎实。
