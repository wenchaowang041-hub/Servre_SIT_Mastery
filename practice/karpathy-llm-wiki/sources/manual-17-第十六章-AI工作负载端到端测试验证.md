# 第十六章 AI 工作负载端到端测试验证

## 1. 模块概述与重要性

AI 端到端测试是整套手册的最后落点。  
前面所有模块的意义，最终都要回到一个问题：

这台机器或这套平台，能不能稳定承载真实 AI 训练和推理业务。

端到端测试最容易犯的错误，是只看模型能不能跑起来。  
真正需要回答的是：

- 吞吐是否稳定
- 时延是否稳定
- 加速器利用率是否合理
- Host CPU 是否成为瓶颈
- 网络和存储是否拖后腿
- 长时间运行是否报错

## 3. 基础原理讲解

### 3.1 AI 端到端测试到底在测什么

1. 环境是否可启动
2. 数据是否能供上
3. 加速器是否能高利用率工作
4. 网络和存储是否匹配
5. 长时间运行是否稳定

### 3.2 为什么端到端测试一定要回溯单模块基线

因为端到端性能低时，你不能直接怪模型。  
你必须能回到：

- CPU 基线
- 内存基线
- 网络基线
- 存储基线
- GPU/NPU 基线

## 4. 详细测试用例

### 用例 1：单机推理验证

```bash
python3 run_infer.py --model resnet50 --batch-size 32
```

### 用例 2：单机训练验证

```bash
python3 train.py --epochs 1 --batch-size 64
```

### 用例 3：多机通信验证

```bash
iperf3 -c <peer_ip> -P 8 -t 60
```

### 用例 4：端到端监控采集

步骤：
1. 记录 CPU/内存/网卡/盘/加速器基线
2. 启动训练或推理
3. 每分钟记录吞吐、时延、温度、功耗、错误
4. 保存日志用于领导汇报和问题复盘

## 5. 结果分析与问题诊断方法

性能低时按这个顺序查：

1. 模型和框架配置
2. 加速器利用率
3. Host CPU 供数
4. 网络
5. 存储

## 6. 最佳实践与注意事项

1. 训练和推理必须分开建立基线。
2. 端到端结论必须能回溯到单模块证据链。
3. 不要只报吞吐，必须报时延、利用率、错误、温度和功耗。

## 7. 进阶内容（高阶测试技巧、自动化思路、多厂商混配测试等）

建议建立标准化 AI 基线：

- ResNet 类 CV
- BERT/LLaMA 类 NLP
- 推荐模型
- 向量检索或推理服务
