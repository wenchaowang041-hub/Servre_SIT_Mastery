# 新项目整机 SIT 落地流程清单

## 1. 适用场景

本清单适用于公司来了一个新项目后，从拿到 BOM 和借料清单开始，到机器完成组装、点亮、版本校准、设备识别、NPU 使能、基础环境检查为止的整机 SIT 前置工作。

这不是单纯的“跑几个命令”，而是把一台新项目机器从物料状态推进到“具备专项测试前提”的完整流程。

## 2. 整体流程

建议固定按下面顺序推进：

1. 获取项目资料
2. 核对物料与拓扑
3. 整机组装
4. 板卡与固件版本校准
5. 上电点亮
6. BIOS / BMC 升级与基础配置
7. PCIe 设备识别与链路确认
8. NPU 安装与软件栈部署
9. OS 基线检查
10. 判断是否具备进入专项测试条件

## 3. 第一阶段：项目资料准备

在机器组装前，先把资料拿全，不然后面很容易边装边猜。

需要拿到的资料：

- BOM
- 借料清单
- 接线拓扑图
- 版本要求说明
- 兼容性要求
- NPU / 网卡 / RAID / NVMe 等关键器件适配要求

这一阶段的目标不是动手装机，而是先回答两个问题：

- 这台机器理论上应该装成什么样
- 每个关键部件应该用哪个版本

## 4. 第二阶段：物料与硬件核对

料到之后，先做逐项核对，不要直接安装。

重点检查对象：

- 背板
- 扩展板
- switch 板
- 风扇板
- 挂耳板
- 电源板
- BMC 卡
- PSU
- RAID 卡
- OCP 网卡
- IB 卡
- NVMe
- SATA 盘
- memory
- NPU

需要重点记录的信息：

- 型号
- 容量
- 厂商
- SN
- 功耗
- 支持代际

线缆也要单独核对，不要只看能不能插上。

重点线缆类型：

- 电源线
- 低速线
- 高速线

线缆重点记录项：

- 线缆 SN
- 厂商
- 规格
- 对应连接位置

## 5. 第三阶段：板卡版本与安装前校准

在安装前要先确认关键板卡版本，不要等上电后才发现版本不匹配。

重点确认：

- 背板对应版本
- 扩展板对应版本
- switch 板对应版本
- CPLD 版本是否需要烧录到项目指定版本

这一阶段的原则是：

- 先确认版本
- 再烧录版本
- 再安装板卡

否则后面出现设备不识别、链路异常、点亮异常时，很难区分是装配问题还是版本问题。

## 6. 第四阶段：整机组装与点亮

完成版本校准后，按接线拓扑进行整机组装。

组装动作包括：

- 安装各类板卡
- 按拓扑连接电源线
- 按拓扑连接低速线
- 按拓扑连接高速线
- 安装存储、内存、网卡、加速卡
- 检查器件安装方向、插槽位置、锁扣状态

完成后进行上电检查。

这一阶段的目标很明确：

- 机器能正常点亮
- 无明显告警
- 能进入 BIOS / BMC / OS 后续检查流程

## 7. 第五阶段：升级 BIOS 和 BMC

点亮只是开始，不代表版本已经符合项目要求。

需要确认：

- BIOS 当前版本
- BMC 当前版本
- 是否需要升级到项目指定版本

常见动作：

- 升级 BIOS
- 升级 BMC
- 复核升级结果

检查方式：

- `dmidecode -t bios`
- 进入 BIOS 界面查看
- 登录 BMC Web 查看 BMC 版本

## 8. 第六阶段：配置 BMC IP 和 OS IP

版本确认后，先把带外和带内管理链路打通。

需要完成：

- 配置 BMC IP
- 配置 OS IP
- 确认网络口状态正常

常用检查命令：

```bash
ip a
ethtool <网口名>
```

重点看：

- 网口是否 up
- 速率是否正确
- 驱动是否正确
- 链路是否稳定

## 9. 第七阶段：PCIe 设备识别与兼容性确认

这一步是整机 SIT 的关键分水岭。

要检查的不是“系统里有没有设备”，而是：

- 设备有没有被正确识别
- 设备是否和 BMC 适配
- 设备是否支持项目要求的 PCIe 代际

重点确认对象：

- RAID 卡
- OCP 网卡
- IB 卡
- NVMe
- NPU
- 其他 PCIe 卡

检查内容：

- 是否枚举
- 是否识别为正确型号
- 是否跑在预期链路代际
- 是否存在不兼容情况

典型判断逻辑：

- 未识别：先和 BMC / 版本适配关系一起看
- 不支持 Gen5：按项目要求更换为 Gen4 方案或更换兼容卡件

常用命令：

```bash
lspci
lspci -nn
lspci -vv
```

重点看：

- 设备是否存在
- `LnkCap`
- `LnkSta`
- 驱动绑定情况

## 10. 第八阶段：NPU 安装与软件栈部署

如果项目是昇腾或 AI 服务器方向，这一步非常关键。

需要完成：

- 安装 NPU
- 安装驱动和固件
- 安装 Ascend 工具
- 安装 CANN 工具包
- 检查 NPU 是否能被正常识别

检查目标：

- NPU 在 PCIe 层可见
- 驱动加载正常
- 工具链安装正常
- 可以正常查看 NPU 信息

常用检查思路：

- 先看 `lspci`
- 再看驱动和固件
- 再看 Ascend / CANN 工具链
- 最后看 `npu-smi` 类工具输出

## 11. 第九阶段：OS 基线检查

当整机点亮、版本完成、关键设备识别后，开始做 OS 层基线核对。

### 11.1 CPU 检查

```bash
lscpu
```

重点看：

- CPU 逻辑核数
- socket 数
- NUMA 节点数

### 11.2 NUMA 检查

```bash
numactl -H
```

重点看：

- NUMA 节点划分
- 各节点 CPU 分布
- 各节点内存分布

### 11.3 memory 检查

```bash
dmidecode -t memory
free -h
```

重点看：

- memory 型号
- memory 容量
- 当前系统可用容量

### 11.4 块设备检查

```bash
lsblk
```

重点看：

- 当前系统识别到的块设备
- 容量是否符合预期
- 系统盘与数据盘是否清晰

### 11.5 BIOS / BMC 版本检查

```bash
dmidecode -t bios
```

同时结合：

- BIOS 页面
- BMC Web 页面

### 11.6 网口检查

```bash
ip a
ethtool <网口名>
```

重点看：

- 网口信息
- 速率
- 双工模式
- 驱动信息

## 12. 第十阶段：进入专项测试前的放行条件

只有下面这些前提都满足，才建议进入 CPU、内存、存储、网络、NPU、整机稳定性等专项测试：

- BOM 与实物一致
- 借料清单与装机结果一致
- 接线拓扑与现场装配一致
- 关键板卡和 CPLD 已烧录到指定版本
- BIOS / BMC 版本符合项目要求
- BMC IP 和 OS IP 配置完成
- PCIe 关键设备识别完整
- PCIe 代际满足项目要求
- NPU 驱动、固件、Ascend、CANN 安装正常
- CPU / NUMA / memory / 块设备 / 网口检查结果正常

如果这些前提里有任何一个不满足，后面的专项测试结果都要谨慎使用，不能直接下结论。

## 13. 你这类项目的真实工作重点

从你描述的流程看，你的工作重点其实不只是“测试”，而是以下四件事的组合：

1. 项目硬件导入
2. 版本校准与适配确认
3. 整机 bring-up
4. 测试前基线建立

这比单纯会跑命令更接近高级 SIT 工程师的工作方式。

## 14. 建议后续继续沉淀的模板

建议后面继续把下面几类模板单独做出来：

- 新项目装机检查清单
- 版本烧录记录表
- 线缆与部件 SN 记录表
- 点亮失败排查清单
- PCIe 设备识别异常排查清单
- NPU 安装与 CANN 部署检查清单

## 15. 按你当前工作方式整理后的完整表述

如果公司来了一个新项目，我一般会先拿到这个项目对应的 BOM、借料清单、接线拓扑图，以及版本要求和适配要求。料到之后不会直接装机，而是先逐项核对实物和资料是否一致，包括背板、扩展板、switch 板、风扇板、挂耳板、电源板、BMC 卡、PSU、RAID 卡、OCP 网卡、IB 卡、NVMe、SATA、memory、NPU 等关键器件，同时把线缆也一起核对清楚，区分电源线、低速线和高速线，记录它们的 SN、厂商、规格和连接位置。

在正式安装前，我还会先确认背板、扩展板、switch 板这些关键板卡的版本是否符合项目要求，尤其是 CPLD 版本是否需要提前烧录到指定版本。这个步骤很重要，因为如果版本不对，后面即使机器能点亮，也可能出现设备不识别、链路异常或者兼容性问题，到时候很难区分到底是装配问题还是版本问题。

硬件和版本确认完成后，我会按照接线拓扑进行整机组装，把板卡、存储、内存、网卡、NPU 这些器件逐个安装到对应槽位，并按照拓扑连接好电源线、低速线和高速线。装完之后先做上电点亮，确认机器能够正常启动，没有明显告警，并且可以进入 BIOS、BMC 和 OS 的后续检查流程。

机器点亮后，我会先做版本校准，升级 BIOS 和 BMC 到项目指定版本，然后配置好 BMC 的 IP 和 OS 的 IP，把带外和带内管理链路先打通。之后开始检查 PCIe 设备识别情况，确认 RAID 卡、OCP 网卡、IB 卡、NVMe、NPU 等关键设备是否都被系统正常枚举，是否识别为正确型号，是否运行在项目要求的 PCIe 链路代际上。这里不只是看有没有设备，更要看链路状态和适配关系。对于未识别的设备，不能只盯着 OS，要同时结合 BIOS、BMC、CPLD、板卡固件版本、插槽拓扑、供电和平台适配矩阵一起分析。对于标称支持 Gen5 但实际不满足条件的场景，也不能简单理解成“卡不行”，还要进一步确认是设备本身、插槽、riser、retimer、线材还是平台版本导致，如果最终不满足 Gen5 条件，就需要按项目要求切换为 Gen4 方案或者更换兼容器件。

如果是昇腾或 AI 服务器项目，PCIe 层确认之后，我还会继续安装 NPU，部署驱动、固件、Ascend 工具和 CANN 工具包，再逐层检查 NPU 是否正常工作。这个过程通常是先确认 `lspci` 是否能看到 NPU，再确认驱动和固件是否正常加载，然后检查 Ascend 和 CANN 工具链是否安装正确，最后通过 `npu-smi` 一类工具确认能否正常读取 NPU 信息。这样做的目的是把问题定位顺序固定下来，避免把工具链问题误判成硬件问题，或者把硬件枚举问题误判成驱动问题。

在关键设备识别完成后，我会做 OS 层基线检查。CPU 方面会用 `lscpu` 看逻辑核数、socket 数和 NUMA 节点数；NUMA 方面会用 `numactl -H` 看各节点 CPU 和内存的分布是否合理；memory 方面会结合 `dmidecode -t memory`、`free -h` 一起看，其中 `dmidecode -t memory` 主要确认安装了哪些内存条、型号和容量，`free -h` 主要确认 OS 当前可见和可用的内存容量，这两个结果不能混为一谈；块设备方面会用 `lsblk` 看系统当前识别到的盘和容量；BIOS 和 BMC 版本会结合 `dmidecode -t bios`、BIOS 页面和 BMC Web 页面一起确认；网络方面会用 `ip a` 看网口信息，再用 `ethtool <port>` 看速率、双工和链路状态，用 `ethtool -i <port>` 看驱动和固件信息。

等这些检查都完成之后，我才会判断这台机器是否具备进入下一阶段专项测试的条件。也就是说，我的工作不只是把机器装起来和点亮，更重要的是先把 BOM、版本、拓扑、硬件识别、NPU 软件栈、OS 基线这些前提全部校准清楚，再决定是否进入 CPU、内存、存储、网络、NPU 或整机稳定性测试。这样做的目的，是先建立一个可信的整机基线，避免后面的测试结果失真，也避免把环境问题、版本问题或者兼容性问题错误地归因到硬件本身。
