## 先记住一句话

NUMA 不是一个抽象名词，它决定了 CPU、内存、PCIe 设备和业务线程之间的数据路径成本。

## 一、为什么 Day17 要回到 NUMA

你前面已经学过：

- Day13：IRQ、CPU、NUMA 和网卡性能路径
- Day15：分层意识
- Day16：日志和驱动证据链

但这些内容要真正站稳，有一个前提：

你必须把 NUMA 当成服务器结构的底板，而不是补充知识。

因为下面这些东西，最终都会回到 NUMA：

- CPU 绑核
- IRQ 分布
- 网卡性能
- NVMe 路径
- NPU 路径
- 内存访问成本

## 二、NUMA 到底是什么

NUMA 的全称是：

Non-Uniform Memory Access  
非一致性内存访问

它真正想表达的是：

```text
不是所有 CPU 访问所有内存的代价都一样
```

在服务器里，CPU 更适合优先访问离自己更近的那部分内存。  
如果去访问远端节点的内存，通常成本会更高。

所以 NUMA 不是“机器里有几块内存”的概念，而是：

- 哪些 CPU 更接近哪些内存
- 哪些设备更接近哪些 CPU
- 数据路径是不是就近

## 三、为什么服务器里一定要看 NUMA

如果你只看：

- 总 CPU 数
- 总内存容量

那你只能知道：

```text
这台机器一共有多少资源
```

但你不知道：

```text
这些资源彼此离得近不近
```

而服务器性能很多时候恰恰不是输在总量，而是输在路径。

例如：

- CPU 在 NUMA0
- 业务内存主要落在 NUMA3
- 网卡挂在 NUMA1
- IRQ 又打在 NUMA2

那这条路径就会非常分散。

## 四、今天必须看懂的 4 个字段

### 1. node

一个 NUMA 节点。

你可以先粗理解成：

- 一组更接近的 CPU
- 一组更接近的本地内存

### 2. CPU 范围

每个 node 下通常会有一组 CPU 编号。

这表示：

- 哪些逻辑 CPU 归这个 NUMA 节点

### 3. size / free

每个 NUMA 节点本地内存的大小和空闲情况。

这表示：

- 这个节点上有多少本地内存
- 当前剩多少

要注意：

- `numactl -H` 里的 `nodeX size` 更接近各 NUMA 节点的本地内存总量
- `nodeX free` 表示各节点当前空闲情况
- `free -h` 看到的是 OS 视角下整机整体内存状态

所以：

- 所有 `node size` 加起来，大体接近整机总内存
- 但 `free -h` 的 `available` 不等于简单把各节点 `free` 直接相加
- 一个看 NUMA 分布，一个看系统整体可用性

### 4. distance

这是 Day17 很关键但容易被忽略的字段。

它表示：

- 各 NUMA 节点之间访问代价的相对距离

一般你会看到：

- 访问自己节点，distance 最小
- 访问其他节点，distance 更大

这不是“物理米数”，而是性能成本的相对表示。

## 五、核心命令

```bash
lscpu | grep NUMA
numactl --hardware
```

也可以写成：

```bash
numactl -H
```

## 六、你今天应该怎么看输出

### 第1步：先看 NUMA 节点数

```bash
lscpu | grep NUMA
```

你要先知道：

- 这台机器有几个 NUMA node

### 第2步：再看每个 node 的 CPU 范围

```bash
numactl -H
```

重点看：

- node0 cpus
- node1 cpus
- node2 cpus
- node3 cpus

你要搞清楚：

- 哪些 CPU 属于哪个 NUMA

### 第3步：再看每个 node 的内存大小

同样在：

```bash
numactl -H
```

重点看：

- nodeX size
- nodeX free

你要搞清楚：

- 各节点内存是不是大致均衡
- 有没有某个节点明显偏小

如果一台机器：

- NUMA 节点数正常
- 每个节点 CPU 数差不多
- 每个节点内存大小也比较接近

通常说明：

- 整机基础拓扑比较规整
- 资源分布比较均衡
- 后续继续分析设备 NUMA 归属时，基础是可靠的

反过来如果出现明显不均衡，就要怀疑：

- DIMM 插法问题
- 某侧内存缺失
- BIOS 配置影响
- 节点资源暴露异常

### 第4步：最后看 distance

```bash
numactl -H
```

最后通常会有 distance matrix。

你要理解的不是具体数值，而是：

- 自己访问自己最近
- 访问别的节点更贵

常见读法可以先这样理解：

- 对角线最小，表示本地访问
- 小幅增大，表示较近节点
- 明显增大，表示更远路径，很多时候更像跨 socket

例如：

- `0 -> 1 = 12`，通常可理解为 `node0` 和 `node1` 更近
- `0 -> 2 = 35`，通常说明 `node0` 和 `node2` 的路径成本明显更高

注意这里说的是相对路径成本，不是物理距离。

这就是后面讨论跨 NUMA 成本的基础。

## 七、为什么 Day17 会直接影响后面的所有专题

### 1. 对网络

网卡挂在哪个 NUMA、IRQ 打在哪个 CPU、业务线程跑在哪个 node，都会影响吞吐和延迟。

如果：

- 网卡挂在 `NUMA3`
- 业务线程主要跑在 `NUMA0`
- `0 -> 3 distance` 又很高

那就要警惕：

- 路径成本增加
- 吞吐下降
- 延迟升高
- 抖动变大

### 2. 对存储

NVMe 挂在哪个 node，会影响 CPU 处理路径和数据搬运成本。

### 3. 对 NPU / AI 服务器

如果 NPU、CPU、内存不在更优路径上，性能就容易打折。

### 4. 对整机 SIT

很多“性能不达标”问题，本质不是设备坏，而是路径没站对。

## 八、你今天最容易犯的错

### 错误1：把 NUMA 只理解成“本地快、远程慢”

这句话没错，但太浅。

更准确的是：

NUMA 决定整机资源和设备之间的相对路径关系。

### 错误2：只看节点数，不看 CPU 范围

只知道“有 4 个 NUMA”意义不大，关键是每个 node 管哪些 CPU。

### 错误3：只看 CPU，不看内存和 distance

NUMA 不是 CPU 话题，而是 CPU + 内存 + 路径成本的话题。

## 十、面试式自测题

### 题1

为什么服务器里不能只看总 CPU 和总内存？

标准答案方向：

- 因为总量只能说明资源有多少，NUMA 才说明资源之间的路径成本

### 题2

`numactl -H` 里你最该看哪几个字段？

标准答案方向：

- node
- cpus
- size / free
- distance

### 题3

NUMA 为什么会影响网卡、NVMe、NPU 的表现？

标准答案方向：

- 因为设备、CPU、内存、业务线程的数据路径成本不一样

### 题4

为什么不能把 `free -h` 看到的可用内存，直接等同于 `numactl -H` 里某几个 node 的空闲内存？

标准答案方向：

- `free -h` 是 OS 视角下的整体可用内存
- `numactl -H` 是 NUMA 视角下的节点分布
- 两者观察角度不同，不能简单一对一套用
