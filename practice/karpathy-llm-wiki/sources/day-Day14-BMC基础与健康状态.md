# Day14：BMC 基础与健康状态

## 对齐计划表

- 阶段：Phase1-系统
- 周次：W2
- 模块：BMC
- 主题：BMC 基础与健康状态

## 今日目标

- 理解 BMC 在服务器里的角色
- 学会把 BMC 当作正式的硬件健康与日志入口
- 掌握传感器、SEL、带外管理这三个核心概念

## 先记住一句话

BMC 不是“远程开机关机的小网页”，而是整机 SIT 排障时最稳定的硬件观察入口之一。

## 一、什么是 BMC

BMC 可以先简单理解为服务器的带外管理控制器。

它和 OS 不是一层东西。

就算 OS 没起来、业务网口没通、甚至系统卡住了，BMC 往往还能继续提供：

- 设备管理入口
- 传感器状态
- 告警日志
- 电源控制
- 远程控制台

所以你要把它看成“独立于 OS 的硬件观察面”。

## 二、为什么 BMC 对 SIT 很重要

因为很多硬件问题，OS 侧看不全，或者看到的已经是结果，不是原因。

例如：

- 机器过温
- 风扇异常
- PSU 掉电
- 电压异常
- 反复重启
- 板卡掉电告警

这些现象，BMC 往往比 OS 更早、更直接记录下来。

所以做整机 SIT 时，BMC 的价值主要在三件事：

1. 看当前健康状态
2. 看历史异常记录
3. 为后续排障提供时间线证据

## 三、今天要掌握的三个核心概念

### 1. 传感器

传感器就是当前硬件状态的实时读数入口。

常见项目包括：

- 温度
- 电压
- 风扇转速
- 电源状态
- 电流

### 2. SEL

SEL 是 System Event Log，可以理解成 BMC 侧的硬件事件日志。

你可以把它看成：

- 什么时候过温了
- 什么时候掉电了
- 什么时候风扇异常了
- 什么时候发生了硬件告警

### 3. 带外管理

带外管理的意思是：  
不依赖 OS 主业务网络，也能管理服务器。

这就是为什么：

- OS 挂了还能看机器
- 网口还没配好也能先进 BMC
- 现场 bring-up 阶段通常先打通 BMC

## 四、核心命令

```bash
ipmitool sensor
ipmitool sel list
```

如果环境支持，也建议看：

```bash
ipmitool sdr list full
ipmitool mc info
ipmitool fru
```

## 五、今天的标准训练顺序

### 第1步：确认 BMC 是不是可访问

你先要确认的不是传感器，而是：

- BMC IP 是否可达
- Web 是否能打开
- `ipmitool` 是否能取到数据

如果 BMC 自己都不稳定，那后面的健康判断也要打折。

### 第2步：先看当前传感器状态

执行：

```bash
ipmitool sensor
```

重点看：

- 有没有 `ok`
- 有没有 `cr`
- 有没有 `nr`
- 有没有明显超阈值

你今天先不追求把每个字段都背下来，先能分清：

- 正常
- 临界
- 非恢复
- 不可用

### 第3步：再看 SEL 历史日志

执行：

```bash
ipmitool sel list
```

你要看的是：

- 最近有没有异常
- 异常集中发生在什么时候
- 是否和现场故障时间能对上

很多时候当前传感器已经恢复正常，但 SEL 里会留下历史证据。

### 第4步：把当前状态和历史状态结合

这是今天最重要的能力。

只看 `sensor`，你看到的是“现在”。  
只看 `SEL`，你看到的是“过去”。

真正有价值的是把两者拼起来：

- 现在正常，但过去反复过温
- 现在电源正常，但过去有 PSU 掉电
- 现在风扇在转，但过去有转速异常

这才是整机 SIT 的健康判断。

## 六、建议你今天输出什么

按计划表，今天的硬性产出是：

- `BMC 状态表 + SEL 摘录`

你可以直接按下面这个结构写：

### BMC 状态表

| 项目 | 当前状态 | 备注 |
|---|---|---|
| CPU 温度 | 正常/异常 | 是否接近阈值 |
| 风扇 | 正常/异常 | 是否有低转速 |
| PSU | 正常/异常 | 是否掉电 |
| 电压 | 正常/异常 | 是否漂移 |

### SEL 摘录

| 时间 | 事件 | 初步解释 |
|---|---|---|
| 2026-03-30 xx:xx | Fan fault | 风扇可能掉速或连接异常 |
| 2026-03-30 xx:xx | PSU lost | 电源输入异常或接触问题 |

## 七、今天最容易犯的错

### 错误1：把 BMC 只当登录入口

真正有价值的不是登录进去，而是用它看健康和日志。

### 错误2：只看当前传感器，不看 SEL

这样你会漏掉已经发生过、但暂时恢复的异常。

### 错误3：看到告警就直接下结论

告警是线索，不是最终结论。  
你还要结合现场时间、负载、OS 日志和物理状态一起看。

## 九、面试式自测题

### 题1

为什么 BMC 对服务器排障很重要？

标准答案方向：

- 因为它独立于 OS
- 可以提供传感器、SEL、带外管理能力
- 是硬件健康和异常历史的重要证据源

### 题2

为什么不能只看 `ipmitool sensor`？

标准答案方向：

- 因为那只能反映当前状态
- 很多异常是瞬时发生的，需要结合 `SEL` 看历史

### 题3

你今天最该建立的关系是什么？

标准答案方向：

- `当前传感器状态 + SEL 历史日志 + 现场现象`

## 十、今天的补习结论

- BMC 是整机 SIT 的核心观察入口之一。
- 健康判断不能只看当前状态，还要结合历史事件。
- 后面做整机稳定性、过温、掉电、风扇、告警排查时，Day14 会直接成为基础能力。
