# 第五章 网络测试（包括 RoCE、DPDK、高速网卡）

## 1. 模块概述与重要性

服务器网络测试是整机 SIT 里最容易“看起来简单，实际最容易翻车”的模块之一。

为什么这么说：

- `ping` 通不代表网络没问题
- `iperf3` 能跑出带宽不代表业务一定稳定
- 网卡识别正常不代表 PCIe、NUMA、中断、PFC/ECN 都配置正确
- RoCE 没丢包不代表尾时延稳定
- DPDK 能收发包不代表实际零拷贝和队列亲和性已经优化到位

2026 年的服务器网络测试已经不是传统“千兆/万兆口验收”的思路，而是进入以下场景：

1. AI 服务器和训练集群
   - 关注 RoCE、拥塞控制、PFC/ECN、400G/800G 网络
2. 云与虚拟化平台
   - 关注 SR-IOV、VF、overlay、SmartNIC/DPU 协同
3. 存储网络与 NVMe-oF
   - 关注低时延、稳定吞吐、丢包与重传
4. DPDK/用户态收发场景
   - 关注 CPU 绑核、RSS、多队列、NUMA、本地内存

网络测试真正值钱的地方，不是“跑几个命令”，而是你能不能判断：

- 这条链路为什么跑不满
- 吞吐不稳是网卡问题、交换机问题，还是 CPU/NUMA 问题
- RoCE 时延波动是无损配置问题，还是队列/中断问题
- DPDK 收包抖动是绑核不合理，还是 hugepage/NUMA 没配置对

高级工程师做网络测试，必须同时有三个视角：

1. 主机视角
2. 网卡视角
3. 网络面视角

## 3. 基础原理讲解（通俗易懂）

### 3.1 网络测试到底在测什么

服务器网络测试至少要回答这几件事：

1. 链路是否正确建立
2. 驱动、固件、速率、双工、FEC 是否正确
3. 吞吐是否达标
4. 时延和尾时延是否稳定
5. 丢包、重传、错误包是否可接受
6. 多队列、中断、NUMA、CPU 亲和性是否合理
7. RoCE/DPDK/SR-IOV 是否真的按设计生效

### 3.2 为什么“带宽不满”常常不是网卡本身的问题

因为网络性能是整机协同结果。

举几个典型例子：

- 网卡在 NUMA 节点 1，但业务进程绑在节点 0，吞吐会掉
- 中断全打在少数几个核上，CPU 看起来没满，但网络上不去
- PCIe 只跑到 x8，网卡再强也跑不满
- RoCE 交换机没开 PFC/ECN，业务会抖动甚至重传
- DPDK 队列数和绑核策略错位，延迟会很差

所以网络测试必须同时看：

- `ethtool`
- `lspci`
- CPU/NUMA
- 交换机和链路配置

### 3.3 网络测试里的几个关键概念

#### 3.3.1 RSS、多队列、中断亲和性

- RSS：把流量散到多个队列
- 多队列：让多个 CPU 并行处理网络数据
- 中断亲和性：让队列对应的中断跑在合适 CPU 上

如果这三件事不对齐，网卡即使是 100G/200G/400G，也很难跑满。

#### 3.3.2 RoCE

RoCE 不是“装了 rdma-core 就完事”，它要求：

- 主机侧 RDMA 能力正常
- 交换机侧无损配置合理
- PFC/ECN 参数正确
- 网卡、驱动、固件版本匹配

RoCE 测试不光要看带宽，还要看尾时延和重传风险。

#### 3.3.3 DPDK

DPDK 重点不是“能不能启动 testpmd”，而是：

- hugepage 是否配置正确
- 设备是否正确绑定到用户态驱动
- CPU 和 NUMA 是否本地化
- 队列数和 lcore 分配是否合理

## 4. 详细测试用例

### 用例 1：基础链路、驱动、固件与统计信息检查

#### 测试目的

确认网卡是否识别正常、链路是否 up、协商速率是否正确、驱动与固件是否匹配。

#### 前置条件

- 网线或光模块已经正确接入
- 对端设备正常

#### 所需工具

- `ip`
- `ethtool`
- `lspci`

#### 测试步骤

1. 查看网卡接口是否存在。
2. 查看链路状态和协商速率。
3. 查看驱动和固件版本。
4. 查看关键统计信息，确认没有明显错误包。

#### 完整测试命令

```bash
ip link show
ethtool eth0
ethtool -i eth0
ethtool -S eth0
lspci | grep -i -E "ethernet|network"
```

#### 重点输出字段表

| 字段 | 含义 | 正常范围 | 常见异常及处理方法 |
|---|---|---|---|
| `Link detected` | 链路是否连通 | `yes` | `no`：查线缆、光模块、对端端口 |
| `Speed` | 协商速率 | 与网卡/交换机设计一致 | 降速：查模块、FEC、交换机端配置 |
| `Duplex` | 双工模式 | Full | Half 或异常：排查自动协商 |
| `driver` | 驱动名称 | 与厂商设备匹配 | 异常：驱动未安装或版本错误 |
| `firmware-version` | 固件版本 | 在项目矩阵内 | 偏旧：升级固件验证 |
| `rx_errors/tx_errors` | 收发错误包 | 应接近 0 | 增长：查链路质量和对端 |

### 用例 2：吞吐测试与多流并发验证

#### 测试目的

验证网卡在单流和多流场景下的吞吐能力，识别 CPU、中断、NUMA 和 PCIe 相关瓶颈。

#### 前置条件

- 对端已启动 `iperf3 -s`

#### 所需工具

- `iperf3`
- `sar`
- `mpstat`

#### 测试步骤

1. 跑单流吞吐。
2. 跑多流吞吐。
3. 观察 CPU 占用与重传情况。
4. 对比不同线程数下的带宽变化。

#### 完整测试命令

```bash
iperf3 -c <server_ip> -P 1 -t 60
iperf3 -c <server_ip> -P 8 -t 60
iperf3 -c <server_ip> -P 16 -t 60
mpstat -P ALL 1 10
```

#### 重点输出字段表

| 字段 | 含义 | 正常范围 | 常见异常及处理方法 |
|---|---|---|---|
| `sender/receiver bitrate` | 实际带宽 | 接近链路能力 | 偏低：查 CPU、中断、NUMA、PCIe |
| `Retr` | TCP 重传 | 越低越好 | 偏高：查网络拥塞或链路质量 |
| `CPU %usr/%soft` | CPU 使用情况 | 随网络负载上升 | `softirq` 偏高：查 RSS/中断分布 |

### 用例 3：RSS、多队列与中断亲和性检查

#### 测试目的

确认多队列是否开启，RSS 是否正常工作，中断是否分布在合理的 CPU 上。

#### 所需工具

- `ethtool`
- `cat`
- `grep`

#### 完整测试命令

```bash
ethtool -l eth0
ethtool -x eth0 2>/dev/null || true
cat /proc/interrupts | grep -i eth0
```

#### 重点输出字段表

| 字段 | 含义 | 正常范围 | 常见异常及处理方法 |
|---|---|---|---|
| `Combined` / `RX/TX` queues | 队列数 | 应符合网卡能力和场景需求 | 队列数太少：查驱动/参数 |
| `/proc/interrupts` | 中断分布 | 应分散到多个核心 | 集中在少数核：手工调整亲和性 |

### 用例 4：RoCE 基线验证

#### 测试目的

验证 RDMA 设备识别、RoCE 基础连通性和主机侧配置状态。

#### 前置条件

- 已安装 `rdma-core`
- 交换机与主机 RoCE 配置已完成

#### 所需工具

- `ibv_devinfo`
- `rdma`
- `ethtool`

#### 完整测试命令

```bash
ibv_devinfo
rdma link
ethtool -S eth0
dmesg -T | egrep -i "rdma|roce|mlx|ib"
```

#### 重点输出字段表

| 字段 | 含义 | 正常范围 | 常见异常及处理方法 |
|---|---|---|---|
| `transport` | RDMA 传输类型 | 应为 RoCE 目标模式 | 不符：查驱动和设备能力 |
| `active_mtu` | 活动 MTU | 与网络规划一致 | 偏低：查对端 MTU |
| RDMA 错误统计 | RDMA 异常 | 不应持续增长 | 增长：查 PFC/ECN/交换机 |

### 用例 5：DPDK 绑定与 `testpmd` 收发验证

#### 测试目的

验证用户态收发路径是否正常，并观察队列、CPU、NUMA 配置是否合理。

#### 前置条件

- 已安装 DPDK
- hugepage 已配置

#### 所需工具

- `dpdk-devbind.py`
- `testpmd`

#### 完整测试命令

```bash
dpdk-devbind.py --status
testpmd -l 0-7 -n 4 --socket-mem=2048,2048 -- -i --forward-mode=io
```

#### 重点输出字段表

| 字段 | 含义 | 正常范围 | 常见异常及处理方法 |
|---|---|---|---|
| `driver` | 当前绑定驱动 | 应为用户态驱动或目标驱动 | 绑定失败：查 IOMMU 和驱动 |
| `RX-packets/TX-packets` | 收发包数 | 应持续增长 | 无增长：查端口状态或转发模式 |
| `socket-mem` | NUMA hugepage 分配 | 应与设备所在 NUMA 匹配 | 不匹配：性能明显受损 |

## 5. 结果分析与问题诊断方法

### 5.1 网络问题的分析顺序

1. 先看链路是否 up
2. 再看协商速率与 FEC
3. 再看驱动和固件
4. 再看 PCIe 和 NUMA
5. 再看 RSS、队列和中断
6. 最后看交换机和网络面

### 5.2 常见异常一：链路起来了，但吞吐跑不满

常见原因：

- CPU 不在本地节点
- 中断打偏
- PCIe 降宽
- 多流数不够

### 5.3 常见异常二：RoCE 能跑，但时延抖动很大

常见原因：

- PFC/ECN 不合理
- 交换机队列拥塞
- CPU 绑核和业务线程冲突

### 5.4 常见异常三：DPDK 能启动但吞吐不稳

常见原因：

- hugepage 不在本地 NUMA
- lcore 绑定错误
- 队列数和流量模型不匹配

## 6. 最佳实践与注意事项

1. 网络测试必须记录网卡所在 NUMA 节点。
2. 先做基础链路检查，再做带宽，再做 RoCE/DPDK。
3. 对 100G 以上网卡，CPU、中断和 NUMA 影响会非常大。
4. 不要只报告带宽，要报告重传、错误包和 CPU 代价。
