# 整机 SIT 综合排障 SOP：识别 / 性能 / 健康三分法

## 1. 适用场景

本 SOP 适用于服务器整机 SIT 现场，面对以下三类高频问题时，快速建立统一排障顺序：

- 设备识别异常
- 性能异常
- 机器状态异常

它的目标不是“多记几个命令”，而是建立一套稳定的排障框架，避免一上来就把所有问题都归成“硬件坏了”。

## 2. 核心原则

排障时先不要急着猜根因，先做三件事：

1. 先分问题类型
2. 再分问题层级
3. 先找证据，再下结论

可以压缩成一句话：

```text
先分类型 -> 再分层 -> 再找证据 -> 最后下结论
```

## 2.1 补充备注：这份 SOP 还必须带上的 6 个现场意识

前面的主框架已经够用，但现场真正稳定的排障，还必须补上下面这些意识，否则很容易出现“顺序对了，但结论还是不稳”的问题。

### 补充1：先确认基线，不要脱离预期谈异常

现场排障前，先确认下面这些“基线信息”是否清楚：

- 送测 BOM
- 借料清单
- 接线拓扑
- BIOS / BMC / FW / CPLD 目标版本
- OS / kernel / driver 目标版本
- 正常机器或历史正常版本的对照结果

如果这些基线本身不清楚，就算你看到了异常，也很难判断它是真的异常，还是本来就不在预期内。

### 补充2：先确认问题能否稳定复现

很多问题不是“有没有”，而是“在什么条件下出现”。

排障时建议先记清楚：

- 上电即出现，还是压测后出现
- 空载出现，还是高负载出现
- 必现、偶现，还是长时间后才出现
- 换槽位、换线、换卡后是否还出现

这一步的价值是把“随机现象”变成“有条件的现象”，后面才能做有效定位。

### 补充3：原始证据先保留，再做调整

这是非常容易漏掉的一点。

在做下面这些动作前，优先保留原始证据：

- 升级 BIOS / BMC / FW
- 换驱动
- 改 BIOS 设置
- 换槽位
- 换线
- 重新装卡

建议先保留：

- `dmesg`
- `journalctl -k`
- `ipmitool sensor`
- `ipmitool sel list`
- `lspci -vv`
- `lsblk`
- `ethtool`
- 当前版本信息

否则一旦你改完环境，原始问题现场就丢了。

### 补充4：版本问题优先用“矩阵思维”看，不要单点看

现场很多问题不是单一版本问题，而是“组合不匹配”。

建议始终把版本按矩阵看：

```text
BIOS
+ BMC
+ Device FW
+ CPLD
+ Driver
+ Kernel
+ OS
```

也就是说：

- 单看 BIOS 正常，不代表组合正常
- 单看驱动版本正确，不代表和当前 kernel 匹配
- 单看 NPU FW 对，不代表和 CANN 工具链匹配

### 补充5：性能异常一定要和“健康状态”交叉验证

这也是目前 SOP 里需要强调的一点。

不要把性能问题和健康问题完全分开。

例如：

- 吞吐下降，可能不是调度问题，而是热降频
- 延迟变高，可能不是 IRQ 问题，而是 PSU/风扇异常导致
- NPU 掉速，可能不是框架问题，而是温度或功耗墙触发

所以性能排障时，建议至少补看一轮：

- `ipmitool sensor`
- `ipmitool sel list`
- BMC Web / Redfish

### 补充6：现场结论必须分“临时判断”和“最终结论”

很多现场排障最大的问题不是不会查，而是过早下最终结论。

建议把结论固定分成两层：

- 临时判断：当前最可能方向是什么
- 最终结论：证据闭环后确认的根因是什么

例如：

- 临时判断：更偏驱动未加载，不像硬件本体故障
- 最终结论：驱动版本与当前 kernel 不兼容，导致模块未加载

这样做可以避免在排障中途把“推测”说成“结论”。

## 3. 第一层：先分问题类型

现场问题建议先分成三类：

### 3.1 设备识别异常

典型现象：

- 设备在工具里看不到
- 系统没有识别出某块卡
- 工具链认不到设备
- 部分功能设备存在，部分不存在

### 3.2 性能异常

典型现象：

- 吞吐不达标
- 延迟偏高
- 抖动大
- 带宽下降
- 队列压力大

### 3.3 机器状态异常

典型现象：

- 机器过温
- 风扇异常
- 电源异常
- 机器不稳定
- 频繁重启
- 历史告警很多

## 4. 第二层：再分问题层级

无论是哪一类问题，都不要直接说“硬件坏了”，先判断它可能属于哪一层：

```text
硬件
-> BIOS / BMC / Firmware / CPLD
-> Driver
-> OS
-> Tool
```

### 4.1 工具层

先确认工具是否使用正确，避免误把工具问题当系统问题。

### 4.2 OS / 内核层

检查当前 OS 版本、内核版本、兼容性和运行状态。

### 4.3 驱动层

检查驱动是否安装、是否加载、是否匹配当前设备和内核。

### 4.4 固件层

检查 BIOS、BMC、设备固件、板卡固件、CPLD 等是否版本对齐。

### 4.5 硬件与拓扑层

检查槽位、接线、板间通信、拓扑、供电和物理连接关系。

## 5. 设备识别异常排查 SOP

### 5.1 第一轮必查项

```bash
lspci
lsmod
dmesg
journalctl -k
```

### 5.2 判断逻辑

- `lspci` 看设备是否在 PCIe 层被枚举
- `lsmod` 看对应驱动模块是否加载
- `dmesg` 看驱动绑定、初始化、报错、reset、timeout
- `journalctl -k` 看启动周期内的内核日志时间线

### 5.3 第二轮补充检查

- 检查拓扑接线
- 检查硬件插槽
- 检查板间通信
- 检查版本是否对齐
- 检查是否存在设备和平台适配问题

### 5.4 这一类问题的典型判断

#### 场景1

- `lspci` 能看到
- `lsmod` 没驱动

优先判断：

- 驱动未安装
- 驱动未加载
- 驱动与内核不匹配

#### 场景2

- `lspci` 看不到

优先判断：

- 插槽 / 接线 / 板卡 / 供电 / 固件 / 拓扑问题

## 6. 性能异常排查 SOP

### 6.1 先分两类

性能问题先分成：

- 链路层问题
- 处理路径问题

### 6.2 链路层重点检查

适合看：

```bash
ethtool <port>
ethtool -i <port>
lspci -vv
```

重点关注：

- 链路是否建立
- 速率是否正确
- 双工是否正常
- PCIe 是否降速降宽
- 对端交换机、网线、光模块是否异常

### 6.3 处理路径重点检查

适合看：

```bash
lscpu
numactl -H
cat /proc/interrupts
top
```

重点关注：

- 网卡 / NVMe / NPU 挂在哪个 NUMA
- IRQ 落在哪些 CPU
- 业务进程跑在哪些 CPU
- 是否存在跨 NUMA / 跨 socket
- 是否有 CPU 热点
- 是否有队列过深、延迟高、带宽低
- 是否有进程抢占资源

### 6.4 这一类问题的核心原则

```text
链路通，不等于性能健康
```

你要建立这条分析链：

```text
设备 NUMA -> IRQ -> CPU -> 业务线程 -> 性能表现
```

## 7. 机器状态异常排查 SOP

### 7.1 优先看 BMC 和硬件健康

适合看：

```bash
ipmitool sensor
ipmitool sel list
```

以及：

- BMC Web
- Redfish

### 7.2 判断逻辑

- `sensor` 看当前状态
- `SEL` 看历史硬件事件
- BMC Web 看模块信息、告警和整体状态

### 7.3 第二轮补充检查

继续结合：

```bash
dmesg
journalctl -k
lsblk
free -h
df -h
```

再核对：

- 当前 OS 内核版本
- BIOS 版本
- BMC 版本
- 各部件固件版本
- CPLD 版本
- 各设备驱动和 FW
- 设备接线
- 内存槽位是否正确
- PCIe 是否降速降宽
- CPU 是否过热

### 7.4 这一类问题的核心原则

```text
当前正常，不代表一直正常
```

如果 `sensor` 正常，但 `SEL` 里连续出现 `Temp high`、`PSU lost`、`Fan fault`，就不能直接开始烧机或压力测试。

## 8. 现场标准话术模板

以后现场排障时，可以直接按下面这套表达：

如果现场出现异常，我会先把问题分成设备识别异常、性能异常和机器状态异常三类，再按分层思路去查。  
如果是设备识别异常，我会先看 PCIe 枚举和驱动层，也就是 `lspci`、`lsmod`、`dmesg`、`journalctl -k`，确认设备是不是在总线上可见、驱动有没有正常加载、日志里有没有初始化失败或报错，同时结合拓扑、接线、插槽和板间通信去判断是不是版本、连接或适配问题。  
如果是性能异常，我会先区分链路层问题和处理路径问题，检查 BIOS 设置、NUMA、CPU、IRQ、延迟、带宽和资源抢占情况，再判断是不是存在跨 NUMA、跨 socket、CPU 热点或 PCIe 链路异常。  
如果是机器状态异常，我会优先看 BMC Web、`ipmitool sensor`、`ipmitool sel list`，确认当前健康状态和历史事件，再结合内核日志、固件版本、驱动、设备状态、接线、内存槽位、PCIe 链路和温度情况去做闭环判断。  
整个过程不会一上来就把问题归成“硬件坏了”，而是先分类型、再分层、再找证据，最后才下结论。

## 9. 最短记忆版

你可以只记下面四句：

1. 先分是识别问题、性能问题，还是健康状态问题。
2. 再分层看，是硬件、固件、驱动、OS 还是工具层。
3. 先用日志、BMC、枚举、驱动、拓扑找证据，不先拍脑袋。
4. 最后才判断是配置、版本、路径还是硬件本体故障。

## 10. 结论

这套 SOP 的价值，不是让你多记几个命令，而是让你在现场问题面前不乱。

真正稳定的 SIT 排障，不是“猜得准”，而是“顺序稳、分层清、证据全”。

最后再补一句：

真正高质量的排障，不只是会查问题，还包括：

- 知道基线是什么
- 知道问题在什么条件下复现
- 知道原始证据要先保留
- 知道版本要按组合看
- 知道临时判断和最终结论要分开

这几条补上之后，这份 SOP 才更接近真实现场可落地的版本。
