# 第十一章 NPU/AI 加速器测试（重点：华为 Ascend 950 系列、npu-smi、CANN 工具链、MindSpore 相关测试）

## 1. 模块概述与重要性

NPU 测试是你后续冲击昇腾/AI 服务器高薪岗位的关键模块。  
这章必须拿下，不然你永远停留在“会跑命令”的层面。

NPU 测试的真正难点在于：

- 卡件识别只是第一步
- 驱动、固件、CANN、框架版本必须成套
- 单卡正常不代表多卡正常
- 多卡正常不代表整机正常
- 整机正常不代表集群训练或推理正常

## 3. 基础原理讲解

### 3.1 NPU 测试到底在测什么

1. NPU 是否识别
2. 健康状态是否正常
3. 驱动和 CANN 是否匹配
4. 单卡算子和样例是否通过
5. 多卡互联是否正常
6. 模型训练/推理是否稳定

### 3.2 为什么 NPU 测试比普通硬件测试更看版本链

因为 NPU 是“硬件 + 驱动 + 工具链 + 框架”的组合系统。  
任何一层版本错配，都可能出现：

- 漏卡
- 工具不可用
- 样例失败
- 性能不稳定
- 长稳中报错

## 4. 详细测试用例

### 用例 1：NPU 识别与健康检查

```bash
npu-smi info
npu-smi topo -t 2>/dev/null || true
```

| 字段 | 含义 | 正常范围 | 异常处理 |
|---|---|---|---|
| `Health` | 健康状态 | 正常 | 异常先停测 |
| `Temperature` | 温度 | 不应接近边界 | 高温查散热 |
| `Power` | 功耗 | 随负载提升 | 过低可能未跑满 |

### 用例 2：驱动与 CANN 版本检查

```bash
cat /usr/local/Ascend/driver/version.info 2>/dev/null || true
cat /usr/local/Ascend/ascend-toolkit/latest/version.info 2>/dev/null || true
```

### 用例 3：基础样例和环境验证

```bash
python3 test_ascend_env.py
```

### 用例 4：MindSpore 或推理样例验证

```bash
python3 run_mindspore_resnet.py
```

### 用例 5：多卡与长稳观察

步骤：
1. 跑单卡样例
2. 跑多卡样例
3. 记录 `npu-smi` 温度、功耗、利用率
4. 检查日志中是否有 device error、timeout、reset

## 5. 结果分析与问题诊断方法

排查顺序：

1. 先看 `npu-smi`
2. 再看驱动/CANN
3. 再看样例
4. 再看多卡互联
5. 最后看端到端模型

## 6. 最佳实践与注意事项

1. 任何 NPU 测试都要写清版本链。
2. 漏卡先查 PCIe/供电/固件，再查工具链。
3. 推理和训练分开建基线。

## 7. 进阶内容（高阶测试技巧、自动化思路、多厂商混配测试等）

建议建立四层基线：

- 单卡健康
- 单机多卡
- 多机通信
- 端到端训练/推理
