# 第十五章 多厂商硬件混配兼容性测试

## 1. 模块概述与重要性

真实项目里，硬件几乎一定是混配的。  
CPU、DIMM、SSD、NIC、GPU/NPU、DPU 来自不同厂商，是常态不是例外。

兼容性测试真正要解决的是：

- 能否识别
- 能否稳定工作
- 性能是否异常退化
- 是否适合量产

## 2. 2026年厂商与主流产品信息对比表

| 方向 | 典型混配组合 | 主要风险 | 重点测试关注点 |
|---|---|---|---|
| CPU + DIMM | 多厂商 DDR5 | 降频、ECC 风险 | 频率、稳定性 |
| CPU + SSD | 多厂商 NVMe | 固件、QoS 差异 | 带宽、尾延迟 |
| CPU + NIC | 多厂商网卡 | 驱动/固件 | 吞吐、SR-IOV |
| CPU + GPU/NPU | 异构加速器 | PCIe/IOMMU/功耗 | 枚举、拓扑、长稳 |

## 3. 基础原理讲解

兼容性不只是“能不能亮机”，而是三层：

1. 功能兼容
2. 性能兼容
3. 长稳兼容

## 4. 详细测试用例

### 用例 1：混配识别基线

```bash
dmidecode -t memory
lspci -nn
lsblk -o NAME,MODEL,SIZE
```

### 用例 2：混配性能对比

```bash
sysbench cpu --cpu-max-prime=20000 --threads=$(nproc) run
fio --name=mix --filename=/data/fio.test --size=20G --rw=randread --bs=4k --iodepth=64 --runtime=60 --time_based --group_reporting
iperf3 -c <server_ip> -P 8 -t 60
```

### 用例 3：混配长稳回归

步骤：
1. 跑 4-8 小时混合负载
2. 每小时记录传感器和错误计数
3. 对比单厂方案基线

## 5. 结果分析与问题诊断方法

先看版本链，再看拓扑，再看性能差异，再看长稳。

## 6. 最佳实践与注意事项

1. 混配测试必须保留批次、固件和版本。
2. 结论要分“可用”和“可量产”。
3. 对国产化替代场景，更要保留长期一致性数据。
