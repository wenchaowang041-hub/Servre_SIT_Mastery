## 前言

服务器硬件整合测试，英文通常称为 Server Integration Test，简称 SIT。很多工程师误以为 SIT 只是“把硬件插上去看看能不能亮机”，这是初级理解。真正的 SIT，是验证一台服务器或者一套计算平台在真实业务约束下，是否能够稳定、持续、可维护地交付目标性能与可靠性。

到了 2026 年，服务器测试的复杂度已经远高于传统双路 x86 时代。原因有四个：

1. 单机硬件复杂度大幅上升。CPU 已经进入高核数时代，Intel Xeon 6、AMD EPYC 9005、华为 Kunpeng 950 都在推进高核心密度与更高内存带宽。
2. 加速器成为主角。NVIDIA Blackwell B200、华为 Ascend 950 系列、各类 DPU/SmartNIC 已经把“整机测试”变成“平台测试”。
3. 互联成为性能上限。很多问题不再是芯片算力不够，而是 NUMA、PCIe、RoCE、UnifiedBus、NVLink、CXL、BMC 策略、散热与供电共同决定系统上限。
4. 客户关注点从“能用”升级到“可规模化交付”。高端客户更关心一致性、长稳、故障恢复、批量部署、固件生命周期、混配兼容性和问题闭环效率。

因此，这本手册不是一本“命令大全”，而是一套面向工程现场的整机测试方法论。目标读者包括：

- 有一定服务器测试基础，希望系统化补齐原理和实战能力的测试工程师
- 正在从硬件测试、系统测试、平台验证向高级工程师升级的工程师
- 需要参与华为鲲鹏/昇腾、Intel/AMD/NVIDIA 平台交付、认证、适配和量产导入的工程师
- 需要编写用例、自动化脚本、测试报告和问题分析报告的工程师

本手册强调三个原则：

- 先建立整机视角，再看单点器件
- 先建立证据链，再下结论
- 先定义可复现步骤，再谈经验判断

你在后续章节中会看到，每个模块都按照统一结构展开：产品对比、原理解释、测试用例、字段解释、问题定位、最佳实践、进阶内容。这样做的目的，是让你不仅知道“怎么测”，还知道“为什么这么测”、“异常意味着什么”、“如何形成可交付的专业结论”。

## 手册使用说明

建议采用“三层学习法”：

1. 第一遍先通读目录、原理和用例流程，建立全局框架。
2. 第二遍按你的实际工作模块深入，边读边在测试机上复现命令与输出。
3. 第三遍把本手册中的步骤沉淀为你自己的 checklist、脚本和报告模板。

推荐学习顺序：

2. 再掌握 CPU、BIOS、内存、网络、存储四大基础模块
3. 然后进入 RAID、电源散热、PCIe、固件/BMC 等平台模块
4. 再学习 GPU、NPU、DPU 等加速模块
5. 最后进入长稳、混配、端到端 AI 工作负载验证

## 测试环境准备

### 1. 模块概述与重要性

测试环境准备不是辅助工作，而是整套 SIT 的起点。如果环境准备不合格，后续任何结论都可能失真。最典型的错误包括：

- 驱动版本与固件版本不匹配，导致性能数据无效
- BIOS 默认值不适合高性能模式，导致 CPU/NPU/GPU 没有跑满
- 机房供电和散热条件不满足，导致烧机阶段频繁降频或保护
- 工具版本不统一，导致横向对比不可复现
- 记录模板缺失，导致问题无法闭环和复盘

所以，资深工程师与普通工程师的差距，往往不是命令记得更多，而是是否能在测试开始前，把环境、版本、方法、记录方式一次性拉齐。

### 3. 基础原理讲解（通俗易懂）

#### 3.1 什么叫“合格的测试环境”

合格的测试环境，至少要同时满足五件事：

1. 硬件配置清楚：CPU、DIMM、盘、网卡、GPU/NPU、DPU、PSU、风扇、PCIe 卡件型号明确
2. 固件版本清楚：BIOS、BMC、CPLD、NIC FW、RAID FW、SSD FW、GPU/NPU FW 明确
3. 软件栈清楚：OS、Kernel、驱动、库、工具链、容器镜像明确
4. 供电散热清楚：PDU、电源冗余、机房温度、风道、液冷/风冷条件明确
5. 记录规则清楚：日志保存位置、命名规范、输出字段、问题单模板明确

如果上面五件事有一件不清楚，测试结论就不可靠。

#### 3.2 为什么 2026 年更强调“整机视角”

以前做双路服务器，很多问题可以直接归因到单个部件。但 2026 年主流平台越来越像“一个系统工程”。

比如一台 AI 服务器训练吞吐上不去，原因可能是：

- CPU NUMA 绑核错了
- PCIe 训练卡没有跑在预期 lane 宽度
- 网卡 PFC/ECN 没调好导致 RoCE 丢包重传
- BMC 风扇策略保守导致加速卡降频
- SSD 盘阵没有达到供数带宽
- 驱动与固件版本不匹配
- 机柜供电或温度边界导致整机进入保护模式

所以高级工程师不是“只看某一个测试工具”，而是能把所有部件串起来看。

#### 3.3 测试环境准备的三层模型

建议把测试环境拆成三层：

- 第一层：静态清单层
  目标是确认“我手上到底是什么机器”
- 第二层：运行基线层
  目标是确认“系统是否已经在正确模式下运行”
- 第三层：验证执行层
  目标是确认“这台机器是否适合开始正式测试”

举例：

- 静态清单层要完成 `dmidecode`、`lshw`、`lspci`、`lsblk`、`ipmitool fru` 等采集
- 运行基线层要完成 BIOS 模式、功耗模式、温度、驱动、服务状态、NUMA 拓扑确认
- 验证执行层要完成样例压测、链路速率确认、错误计数检查、日志路径准备

### 4. 详细测试用例

#### 用例 1：测试前整机基线采集

##### 测试目的

在开始任何专项测试前，完成整机静态与运行基线采集，形成可追溯环境快照。

##### 前置条件

- 服务器已正常上电并进入操作系统
- 拥有 root 或 sudo 权限
- 已安装基础工具：`dmidecode`、`lshw`、`ipmitool`、`pciutils`、`util-linux`

##### 所需工具

- `dmidecode`
- `lshw`
- `ipmitool`
- `lspci`
- `lsblk`
- `numactl`
- `uname`

##### 测试步骤

1. 登录系统并确认主机名、时间、内核版本。
2. 创建统一日志目录，按日期与主机名保存输出。
3. 采集 BIOS、主板、CPU、内存、FRU、传感器、PCIe、块设备、NUMA 拓扑。
4. 记录当前 OS、Kernel、驱动、BMC 版本。
5. 检查输出中是否存在明显异常字段，如 Unknown、No Reading、Degraded、CRC Error、Width x8 instead of x16。

##### 完整测试命令

```bash
TS=$(date +%F_%H%M%S)
HOST=$(hostname)
OUT=/tmp/sit_baseline_${HOST}_${TS}
mkdir -p "${OUT}"

uname -a > "${OUT}/01_uname.txt"
cat /etc/os-release > "${OUT}/02_os_release.txt"
lscpu > "${OUT}/03_lscpu.txt"
numactl -H > "${OUT}/04_numa.txt"
dmidecode > "${OUT}/05_dmidecode.txt"
lshw -quiet > "${OUT}/06_lshw.txt"
lspci -vv > "${OUT}/07_lspci_vv.txt"
lsblk -o NAME,MODEL,SIZE,ROTA,TYPE,MOUNTPOINT > "${OUT}/08_lsblk.txt"
ipmitool mc info > "${OUT}/09_ipmi_mc.txt"
ipmitool fru > "${OUT}/10_ipmi_fru.txt"
ipmitool sdr list full > "${OUT}/11_ipmi_sdr_full.txt"
ipmitool sensor > "${OUT}/12_ipmi_sensor.txt"

echo "baseline saved to ${OUT}"
```

##### 关键参数解析

| 参数/命令 | 含义 | 为什么要采集 |
|---|---|---|
| `uname -a` | 内核与架构信息 | 确认 OS 内核基线 |
| `lscpu` | CPU 拓扑 | 校验 socket/core/thread 是否符合规格 |
| `numactl -H` | NUMA 分布 | 后续 CPU、内存、GPU/NPU 测试的绑定依据 |
| `dmidecode` | SMBIOS/硬件清单 | 主板、BIOS、内存槽位和部件标识 |
| `lshw` | 设备树信息 | 辅助定位设备归属与层级 |
| `lspci -vv` | PCIe 详细链路信息 | 查 lane、速率、AER、ASPM、链路异常 |
| `ipmitool sensor` | 传感器概览 | 温度、电压、风扇、PSU 状态 |

##### 重点输出字段表

| 字段 | 含义 | 正常范围 | 常见异常及处理方法 |
|---|---|---|---|
| `CPU(s)` | 总逻辑核数 | 应与产品规格一致 | 少核：查 BIOS、SMT、CPU 插槽识别 |
| `NUMA node(s)` | NUMA 节点数 | 应与平台设计一致 | 节点异常：查 BIOS SNC/NPS、固件设置 |
| `Speed` / `Width` | PCIe 速率与 lane 宽度 | 应与卡件和槽位设计匹配 | x16 卡只跑到 x8：重新插卡、查 riser、查 BIOS bifurcation |
| `Reading` / `Status` | 传感器读数与状态 | 应为正常范围，无 critical | 温度过高：查风扇策略、风道、环境温度 |
| `Product Name` / `Version` | FRU/主板/BIOS 标识 | 与测试配置单一致 | 不一致：禁止继续对比测试，先校准配置 |

#### 用例 2：测试环境一致性检查

##### 测试目的

确认软件栈与测试工具版本一致，避免因为工具差异导致结果不可比。

##### 前置条件

- 基线采集完成
- 系统可联网或已有离线安装包

##### 所需工具

- `rpm` 或 `dpkg`
- `modinfo`
- `ethtool`
- `fio`

### 5. 结果分析与问题诊断方法

环境准备阶段常见问题与诊断顺序如下：

#### 5.1 设备信息与配置单不一致

诊断顺序：

1. 先看 `dmidecode` 和 `ipmitool fru`
2. 再看板卡实物标签与槽位
3. 再核对采购 BOM、送测清单、SN 绑定关系
4. 最后确认是否存在替换件、工程样机或临时借测卡件

原则：配置不一致时，不要急着跑性能测试。先保证“测的就是你以为的那台机器”。

#### 5.2 PCIe 宽度/速率异常

诊断顺序：

1. `lspci -vv` 看 `LnkCap` 和 `LnkSta`
2. 确认卡插在正确槽位
3. 确认 riser/bifurcation 设置
4. 查 BIOS 中 PCIe speed 是否固定或降级
5. 查是否存在线材、转接板或背板问题

#### 5.3 传感器读数异常

诊断顺序：

1. `ipmitool sensor` 先确认异常项目
2. `ipmitool sdr list full` 看阈值和状态
3. 核查风扇、PSU、温度探头、机箱盖状态
4. 结合机房温度、电源模式、BMC 风扇策略判断

#### 5.4 软件栈版本不一致

诊断顺序：

1. 先看内核版本
2. 再看驱动版本
3. 再看固件版本
4. 最后看工具链版本与容器镜像版本

原则：高级工程师做结论前一定先写清楚版本链路。没有版本链路的结论，不具备复现价值。

### 6. 最佳实践与注意事项

1. 所有正式测试前，先保存一份完整基线快照。
2. 同一个项目内，尽量统一 OS、Kernel、驱动、固件、工具版本。
3. 测试目录命名要包含主机名和时间戳。
4. 任何异常先截图/留日志，再重启或改配置。
5. 做多机对比时，优先确保环境一致，再比较性能差异。
6. 不要把厂商宣传峰值直接当成验收值，必须基于测试方法、精度、负载和场景解释。
7. 对华为 Ascend、Kunpeng 平台，要特别重视 BMC、固件、驱动、CANN 版本的匹配关系。
8. 对 NVIDIA/AMD/Intel 平台，要特别注意 BIOS 电源策略、内存配置、PCIe lane 分配和驱动兼容矩阵。

### 7. 进阶内容（高阶测试技巧、自动化思路、多厂商混配测试等）

#### 7.1 自动化基线采集思路

环境准备最适合最先做自动化。建议你把以下动作写成统一脚本：

- 基线信息采集
- 版本矩阵采集
- PCIe 链路状态检查
- 传感器异常检查
- CPU/内存/盘/网快速烟测

脚本输出建议统一为：

- `summary.md`：给领导和项目经理看
- `raw/`：保留原始输出
- `json/`：给后续自动化比对系统使用

#### 7.2 多厂商平台准备差异

- 华为平台：更关注 Kunpeng/Ascend 配套软件栈、BMC、平台化管理工具、CANN 与驱动匹配
- Intel 平台：更关注 BIOS 电源模式、SNC、内存模式、UPI 与 PCIe 资源划分
- AMD 平台：更关注 NPS、SMT、CCD/CCX 特性、IOMMU、CXL/PCIe lane 规划
- NVIDIA 平台：更关注 GPU 驱动、Fabric Manager、NVLink/NVSwitch、IB/BlueField、功耗散热

#### 7.3 从环境准备就开始培养高级工程师习惯

高级工程师和普通工程师最大的区别，在这一章就已经体现：

- 普通工程师：跑完命令就结束
- 高级工程师：会形成“环境基线 -> 版本矩阵 -> 风险列表 -> 后续测试建议”的完整闭环

建议你从现在开始，把每一次环境准备都输出成标准四段：

1. 当前配置
2. 发现问题
3. 风险判断
4. 后续建议
