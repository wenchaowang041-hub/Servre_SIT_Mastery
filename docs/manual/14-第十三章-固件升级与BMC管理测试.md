# 第十三章 固件升级与 BMC 管理测试

## 1. 模块概述与重要性

固件升级与 BMC 管理测试，决定的是服务器“能不能被持续管理、持续升级、持续交付”。

很多现场故障并不是硬件坏，而是：

- BIOS 升级后配置丢失
- BMC 升级后传感器异常
- 网卡或 SSD 固件升级后性能回退
- 升级链路失败导致批量机器状态不一致

高级工程师做固件/BMC 测试，核心不是“会升级”，而是：

1. 升级前能不能做完整基线
2. 升级过程能不能可控
3. 升级后能不能快速回归
4. 批量升级能不能保持一致性

## 3. 基础原理讲解

### 3.1 为什么固件链必须整体看

因为一台服务器至少有这些版本链：

- BIOS
- BMC
- CPLD
- NIC Firmware
- SSD Firmware
- GPU/NPU Firmware

只看 BIOS 不够，只看 BMC 也不够。  
真正可交付的测试结论必须写清整条版本链。

### 3.2 为什么升级测试本质上是“变更管理测试”

升级不是单点动作，而是一套变更过程：

1. 升级前采集
2. 升级执行
3. 升级后校验
4. 问题回滚

## 4. 详细测试用例

### 用例 1：版本基线采集

```bash
dmidecode -t bios
ipmitool mc info
ipmitool fru
ethtool -i eth0
nvme list 2>/dev/null || true
```

### 用例 2：BMC 基础功能验证

```bash
ipmitool sensor
ipmitool sel list
ipmitool chassis status
```

### 用例 3：带外电源控制与远程重启验证

```bash
ipmitool chassis power status
ipmitool chassis power cycle
```

### 用例 4：升级后最小回归

#### 测试步骤

1. 升级前保存版本和配置。
2. 升级目标固件。
3. 核对新版本是否生效。
4. 回归检查：
   - 能否正常启动
   - 传感器是否正常
   - 关键设备是否识别
   - 日志中有无新增异常

## 5. 结果分析与问题诊断方法

先区分：

- 升级失败
- 升级成功但配置丢失
- 升级成功但功能回退

## 6. 最佳实践与注意事项

1. 升级前必须保留原始版本和配置基线。
2. 升级后必须做最小回归，不要只看版本号。
3. 批量升级要有失败节点标记和回滚策略。
