# 第六章 存储测试（SATA、SAS、NVMe、企业级 SSD）

## 1. 模块概述与重要性

存储测试在服务器 SIT 中承担两种角色：

1. 验证设备本身是否可靠
2. 验证它是否能支撑业务的数据供给需求

对普通通算服务器来说，存储是系统盘、数据盘和日志盘。  
对 AI 服务器来说，存储经常直接决定数据供给速度和任务启动效率。  
对数据库服务器来说，存储的尾延迟和稳定性往往比峰值吞吐更重要。

所以存储测试不能只看“跑分高不高”，而要看：

- 识别是否完整
- 健康状态是否正常
- 顺序吞吐是否达标
- 随机读写是否达标
- 尾延迟和 QoS 是否稳定
- 长稳阶段是否掉速、报错、掉盘
- 固件、PCIe、HBA/RAID、文件系统是否形成了瓶颈

## 3. 基础原理讲解

### 3.1 存储测试到底在测什么

存储测试至少要看：

1. 识别层
   - 盘是否识别完整
2. 健康层
   - SMART/NVMe 日志是否健康
3. 性能层
   - BW、IOPS、时延、尾延迟
4. 稳态层
   - 长时间负载后是否掉速
5. 可靠性层
   - 是否 timeout、reset、掉盘、介质错误

### 3.2 为什么企业 SSD 不能只看空盘跑分

因为很多盘在“空盘 + 短时间”场景下会非常漂亮，但一进稳态就会掉速。真正的企业交付更看：

- 稳态表现
- 尾时延
- 长稳写入
- 错误恢复能力

## 4. 详细测试用例

### 用例 1：设备识别与健康检查

#### 测试目的

确认盘型、容量、型号、介质状态、固件版本和健康状态正常。

#### 所需工具

- `lsblk`
- `nvme`
- `smartctl`

#### 完整测试命令

```bash
lsblk -o NAME,MODEL,SIZE,ROTA,TYPE,MOUNTPOINT
nvme list 2>/dev/null || true
smartctl -a /dev/nvme0n1 2>/dev/null || true
```

#### 重点输出字段表

| 字段 | 含义 | 正常范围 | 常见异常及处理方法 |
|---|---|---|---|
| `MODEL` | 盘型号 | 与送测配置一致 | 不一致：核对替换件 |
| `Firmware` | 固件版本 | 在项目矩阵内 | 偏旧：升级验证 |
| `Critical Warning` | NVMe 关键告警 | 应为 0 | 非 0：优先停测分析 |
| `Media and Data Integrity Errors` | 媒体/完整性错误 | 应为 0 | 增长：盘风险高 |

### 用例 2：顺序吞吐测试

#### 测试目的

验证顺序读写带宽，判断盘或阵列的上限能力。

#### 完整测试命令

```bash
fio --name=seqread \
    --filename=/data/fio.test \
    --size=20G \
    --rw=read \
    --bs=1M \
    --iodepth=32 \
    --ioengine=libaio \
    --direct=1 \
    --runtime=60 \
    --time_based \
    --group_reporting
```

#### 参数解析

| 参数 | 含义 | 作用 |
|---|---|---|
| `--rw=read` | 顺序读 | 看上限吞吐 |
| `--bs=1M` | 大块大小 | 更适合顺序带宽 |
| `--iodepth=32` | 队列深度 | 逼近 SSD 吞吐能力 |
| `--direct=1` | 绕过缓存 | 防止假性能 |

#### 重点输出字段表

| 字段 | 含义 | 正常范围 | 异常处理 |
|---|---|---|---|
| `BW` | 带宽 | 接近设计目标 | 偏低：查 PCIe、NUMA、盘状态 |
| `clat` | 完成时延 | 应稳定 | 波动大：查后台 GC 或固件 |

### 用例 3：随机读写与混合负载测试

#### 测试目的

评估数据库、虚拟化、对象存储等典型场景下的随机性能和尾时延。

#### 完整测试命令

```bash
fio --name=randrw \
    --filename=/data/fio.test \
    --size=20G \
    --rw=randrw \
    --rwmixread=70 \
    --bs=4k \
    --iodepth=64 \
    --ioengine=libaio \
    --direct=1 \
    --runtime=120 \
    --time_based \
    --group_reporting
```

#### 重点输出字段表

| 字段 | 含义 | 正常范围 | 异常处理 |
|---|---|---|---|
| `IOPS` | 每秒 IO 数 | 与盘型匹配 | 偏低：查队列、PCIe、CPU 绑核 |
| `slat/clat/lat` | 提交/完成/总时延 | 越稳定越好 | 尾延迟高：查稳态、GC、混配 |
| `read/write mix` | 读写比例 | 与业务一致 | 不匹配：结果参考价值低 |

### 用例 4：长稳写入与错误恢复测试

#### 测试目的

验证盘在长时间写入下是否掉速、报错或 reset。

#### 完整测试命令

```bash
fio --name=soak \
    --filename=/data/fio.test \
    --size=50G \
    --rw=randwrite \
    --bs=4k \
    --iodepth=64 \
    --ioengine=libaio \
    --direct=1 \
    --runtime=4h \
    --time_based \
    --group_reporting

dmesg -T | egrep -i "nvme|scsi|reset|timeout|error"
```

## 5. 结果分析与问题诊断方法

1. 先看健康状态
2. 再看 PCIe 与 HBA/RAID
3. 再看 FIO 参数是否合理
4. 最后看稳态与错误日志

## 6. 最佳实践与注意事项

1. 企业盘测试要区分空盘、预处理、稳态。
2. 不要只报告峰值，要报告稳态和尾延迟。
3. 同一测试必须写清文件系统、裸盘/文件、NUMA 绑定和深度参数。
