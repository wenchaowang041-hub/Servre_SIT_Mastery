## 一、为什么 Day17 要回到 NUMA

你前面已经在接触这些问题：

- 网卡挂在哪个 NUMA
- IRQ 落在哪些 CPU
- 业务线程跑在哪个 node
- NVMe、NPU 路径是否跨 node / 跨 socket

这些问题表面上看分别属于网络、存储、AI 服务器、系统性能，但底层其实都指向同一个东西：

**NUMA 拓扑。**

所以 Day17 不是重复基础概念，而是把 NUMA 从“背过的知识点”变成“后续所有性能分析的坐标系”。

---

## 二、NUMA 到底是什么

NUMA 全称是：

Non-Uniform Memory Access  
非一致性内存访问

它真正想表达的是：

```text
不是所有 CPU 访问所有内存的代价都一样
```

在服务器里，CPU 更适合优先访问离自己更近的本地内存；如果要访问更远的节点，成本通常就更高。

所以 NUMA 不是单纯在讲“内存”，而是在讲：

- CPU 和内存之间的关系
- 设备和 CPU 之间的关系
- 业务线程和数据路径之间的关系

这也是为什么 NUMA 后面会直接影响：

- 网卡性能
- NVMe 路径
- NPU 路径
- 中断分布
- 业务绑核策略

---

## 三、为什么不能只看总 CPU 和总内存

这个问题是 Day17 的第一核心。

如果你只看：

- 总 CPU 数
- 总内存容量

那你只知道：

```text
这台机器一共有多少资源
```

但你不知道：

```text
这些资源彼此之间的路径成本是多少
```

这就像你知道一座城市总共有多少辆车、多少条路，但你不知道这些路之间近不近、堵不堵、有没有跨城成本。

所以可以这样记：

- 总量：看资源上限
- NUMA：看路径成本

这句话以后你做整机 SIT 会一直用到。

## 四、今天最重要的命令：`numactl -H`

Day17 最关键的命令是：

```bash
numactl -H
```

或者：

```bash
numactl --hardware
```

配合：

```bash
lscpu | grep NUMA
```

一起看，会非常清楚。

## 五、`numactl -H` 里最该看哪 4 类信息

这是 Day17 的第二核心。

### 1. `node`

表示这台机器有几个 NUMA 节点。

例如：

```text
available: 4 nodes (0-3)
```

说明系统当前看到 4 个 NUMA 节点。

### 2. `cpus`

表示每个 node 管哪些逻辑 CPU。

例如：

```text
node 0 cpus: 0 1 2 ... 63
node 1 cpus: 64 65 66 ... 127
```

这一步很关键，因为它直接告诉你：

- 哪些 CPU 属于哪个 NUMA 节点

### 3. `size / free`

表示每个 node 的本地内存大小和空闲内存。

例如：

```text
node 0 size: 515958 MB
node 0 free: 514375 MB
```

这一步能告诉你：

- 每个节点内存是否均衡
- 当前各节点剩余情况如何

这里还要注意一个很容易混淆的问题：

- `numactl -H` 里的 `node size`，更接近 NUMA 视角下各节点的本地内存总量
- `node free`，表示各节点当前空闲情况
- `free -h` 看到的，则是 OS 视角下整机整体内存状态

所以：

- 所有 `node size` 加起来，大体接近系统总内存
- 但 `free -h` 的 `available` 不能简单理解成某几个 node 的 `free` 相加

一个看的是节点分布，一个看的是系统整体可用性。

### 4. `distance`

这是 Day17 最关键也最容易被忽略的一部分。

例如：

```text
node distances:
node   0   1   2   3
  0:  10  12  35  37
  1:  12  10  37  40
  2:  35  37  10  12
  3:  37  40  12  10
```

distance 不是物理长度，而是访问代价的相对表示。

你最该记住的是：

- 自己访问自己，distance 最小
- 和较近节点通信，distance 略高
- 和更远节点通信，distance 明显更高

---

## 六、怎么看这类输出

上面这组数据很典型，可以这样读：

- `node0 -> node0 = 10`，说明本地访问代价最低
- `node0 -> node1 = 12`，说明 node0 和 node1 比较近
- `node0 -> node2 = 35`
- `node0 -> node3 = 37`

这通常意味着：

- `0` 和 `1` 更像同一侧
- `2` 和 `3` 更像另一侧
- `0/1` 到 `2/3` 的路径更远，成本更高

在很多双路服务器里，这种表现就很像：

- 一颗 CPU 下面有两组 NUMA
- 另一颗 CPU 下面有另外两组 NUMA

所以看到 distance matrix 时，你可以先按这个模板理解：

1. 对角线最小，表示本地访问
2. 小幅增大，表示较近路径
3. 明显增大，表示更远路径，往往更像跨 socket

所以如果你看到：

- `0 -> 1 = 12`
- `0 -> 2 = 35`

第一反应就应该是：

- `node0` 和 `node1` 更近
- `node0` 到 `node2` 的路径成本明显更高

这里说的是相对访问代价，不是物理距离。

## 七、为什么 NUMA 会直接影响网卡、NVMe、NPU

Day17 不只是结构课，它会直接影响后面的所有专题。

### 1. 对网卡

如果：

- 网卡挂在 `NUMA3`
- 业务线程跑在 `NUMA0`
- `0 -> 3 distance` 又很高

那你就要警惕：

- 路径成本增加
- 吞吐下降
- 延迟上升
- 抖动变大

### 2. 对 NVMe

NVMe 挂在哪个 NUMA，会影响：

- 哪边 CPU 处理更划算
- IO 路径是否被拉长
- 后续性能是否更稳

### 3. 对 NPU / AI 服务器

如果：

- NPU 在一个 NUMA
- CPU 调度线程在另一个 NUMA
- 数据又在第三个 NUMA

那整条路径就会非常分散，性能自然容易打折。

所以 Day17 的一句话总结就是：

**NUMA 决定的是整条数据路径的成本，而不是单个设备的“好坏”。**

## 八、一个“看起来正常、其实很关键”的现象

如果一台机器：

- 有 4 个 NUMA 节点
- 每个节点 CPU 数差不多
- 每个节点内存大小也差不多

通常说明：

1. NUMA 拓扑比较规整
2. 资源分布比较均衡
3. 后续分析设备归属和性能路径时，基础比较可靠

反过来，如果你看到：

- 某个 node CPU 明显少
- 某个 node 内存明显小
- 某个节点几乎没内存

那就要警惕：

- DIMM 插法问题
- 硬件缺失
- BIOS 配置影响
- 节点资源暴露异常

所以“均衡”不只是看着顺眼，而是说明：

- 这台机器的基础 NUMA 拓扑比较健康
- 后面继续做网卡、NVMe、NPU 的归属分析时，更有参考价值

## 九、今天最容易犯的错

### 错误1：把 NUMA 只理解成“本地快、远程慢”

这句话没错，但太浅。

更准确地说：

**NUMA 决定整机资源和设备之间的相对路径关系。**

### 错误2：只看节点数，不看 CPU 范围

只知道“有几个 NUMA 节点”没多大意义，关键是每个 node 管哪些 CPU。

### 错误3：只看 CPU，不看内存和 distance

NUMA 不是只讲 CPU，它同时讲：

- CPU
- 内存
- 节点距离
- 路径成本

### 错误4：把 NUMA 当成“只有内存优化时才看”

错。  
网卡、NVMe、NPU、IRQ、业务绑核，全都会回到 NUMA。
